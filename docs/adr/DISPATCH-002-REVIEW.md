# DISPATCH-002 Review

## Verdict

NEEDS_FIXES

The three deliverables nominally satisfy the ticket acceptance criteria, but the architecture lock is not ready to advance. The data model is still underspecified for implementation, the save-point lifecycle is missing cleanup semantics, the v1 scope checklist is incomplete relative to the locked roadmap scope, and downstream planning docs still contradict the new ADRs.

## Acceptance Criteria Checklist

- [x] `docs/adr/0003-data-model.md` defines the initial `projects`, `tasks`, `agent_sessions`, and `settings` schema plus locked task-state enums. It also adds `chat_messages` as an early schema table.
- [x] `docs/adr/0004-history-save-points.md` defines `refs/dispatch/*` naming, pre/post/manual/latest save-point rules, and synthetic Dispatch signatures.
- [x] `docs/checklists/v1-scope.md` distinguishes v1 from post-v1 and explicitly defers voice input and Browser.

[docs/adr/0003-data-model.md:56](/home/oso0x/projects/dispatch/docs/adr/0003-data-model.md#L56), [docs/adr/0003-data-model.md:93](/home/oso0x/projects/dispatch/docs/adr/0003-data-model.md#L93), [docs/adr/0003-data-model.md:119](/home/oso0x/projects/dispatch/docs/adr/0003-data-model.md#L119), and [docs/adr/0003-data-model.md:175](/home/oso0x/projects/dispatch/docs/adr/0003-data-model.md#L175) define tables, columns, and some FK behavior, but they do not lock required indexes or DB-enforced constraints. 2. Major: Several foundational column domains remain implicit. [docs/adr/0003-data-model.md:108](/home/oso0x/projects/dispatch/docs/adr/0003-data-model.md#L108) leaves `agent_sessions.status`, `agent_sessions.source`, and `tasks.assigned_agent_mode` as freeform `TEXT` with no canonical value set, unlike the locked task enums. 3. Major: The save-point ADR defines creation and naming but not lifecycle cleanup. No literal `TODO` or `TBD` markers appear in the three reviewed files, but the ambiguities above still need to be closed.

## Fixes Required

- In `docs/adr/0003-data-model.md`, add the required index set for the initial schema and lock whether enum/JSON validity is enforced at the DB layer. At minimum, specify required indexes for project-scoped task queries, session queries, and chat ordering, and either add `CHECK` constraints for locked enums or explicitly state that enforcement is application-only.
- In `docs/adr/0003-data-model.md`, lock canonical value sets for `agent_sessions.status`, `agent_sessions.source`, and `tasks.assigned_agent_mode` so `001_init.sql` does not need to invent new state vocabularies later.
- In `docs/adr/0004-history-save-points.md`, add explicit cleanup semantics: retention policy, whether users can delete/prune save points, how metadata and refs are cleaned up together, and how `latest` is recomputed if the newest save point disappears.
- In `docs/adr/0004-history-save-points.md`, align the git dependency version with the locked roadmap baseline (`git2 0.20.4`) unless the roadmap is intentionally being changed.
- In `docs/checklists/v1-scope.md`, expand the v1 and post-v1 sections to mirror the locked roadmap scope exactly, including the missing v1 items and the missing deferred items.
- Update `ROADMAP.md` and `TICKETS.md` so downstream implementation docs match the new locked decisions on initial schema shape, workflow-state enum, and save-point namespace. Without that propagation, Phase 1 and Phase 6 implementation tickets still point at superseded contracts.
