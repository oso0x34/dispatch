# ADR 0003: Initial Data Model and Persistence Contract

- Status: Accepted
- Date: 2026-03-19
- Deciders: Dispatch Phase 0A architecture lock
- Related: `ROADMAP.md`, `PRD.md`, `TICKETS.md`, `docs/adr/0001-runtime-boundaries.md`, `docs/adr/0002-terminal-lifecycle.md`

## Context

Dispatch needs a durable local model before terminal, task, history, chat, and settings work can start. ADR 0001 already freezes Rust as the owner of SQLite, secrets, and other privileged capabilities. The remaining gap is the concrete v1 schema and the persistence rules that every later ticket must build on.

The planning documents were still carrying drift when DISPATCH-002 started:

- some phase notes listed only `projects`, `tasks`, `agent_sessions`, and `settings`, while the product and runtime-boundary docs already treated chat cache as Rust-owned state
- the roadmap used provisional lowercase workflow-state strings, while the ticket for this turn explicitly calls for a first-class `Blocked` task state
- settings ownership was already frozen in Rust, but the secret precedence and storage exclusions needed to be written down in one place
- SQLite bootstrap behavior such as foreign-key enforcement, WAL mode, and timestamp format had not yet been locked

This ADR freezes the initial schema, enum semantics, timestamp format, and SQLite bootstrap contract so implementation can proceed without re-deciding storage fundamentals.

## Decision

Dispatch uses a Rust-owned SQLite database through `rusqlite 0.39.0`, with SQL migrations stored under `src-tauri/migrations/`. The first schema migration is the canonical starting point for v1 persistence and must create five tables:

- `projects`
- `tasks`
- `agent_sessions`
- `chat_messages`
- `settings`

### Storage and Bootstrap Rules

1. Rust is the only database owner.
   React never writes directly to SQLite and never becomes a second source of truth for durable records.

2. Migrations live under `src-tauri/migrations/`.
   The initial migration is expected to be `001_init.sql`, and all future schema changes extend from that directory.

3. SQLite is opened with these required pragmas:
   - `PRAGMA foreign_keys = ON`
   - `PRAGMA journal_mode = WAL`
   - `PRAGMA journal_size_limit = 67108864`

4. All timestamps are stored as UTC `INTEGER` Unix epoch seconds.
   Dispatch does not store wall-clock timestamps as text, local time, or milliseconds in the canonical schema.

5. Durable identifiers are opaque `TEXT` ids generated by Rust.
   The database schema treats ids as stable strings; the exact generator is an implementation detail as long as ids are unique and transport-safe.

6. Secrets are never stored in SQLite.
   Non-secret settings live in `settings`. Secret values are resolved outside the database via `keyring 3.6.3` with inherited environment-variable fallback.

7. `tauri-plugin-store` is out of scope for v1.
   Dispatch does not persist settings or credentials in a frontend JSON store.

8. Closed storage vocabularies and JSON syntax are enforced at the DB layer.
   `001_init.sql` must use `CHECK` constraints for the locked enum-like columns in this ADR, and it must reject invalid JSON payloads rather than relying on application-only validation.

### Initial Schema Contract

The schema is intentionally small. It covers the surfaces that must exist before feature work begins and avoids history-specific or browser-specific tables that are not part of the Phase 1 foundation.

#### `projects`

Each project row represents one registered project root that Dispatch is allowed to operate on.

Required columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | Stable opaque project id |
| `name` | `TEXT NOT NULL` | User-facing display name |
| `root_path` | `TEXT NOT NULL UNIQUE` | Canonical absolute project root path |
| `created_at` | `INTEGER NOT NULL` | UTC Unix epoch seconds |
| `updated_at` | `INTEGER NOT NULL` | UTC Unix epoch seconds |
| `last_opened_at` | `INTEGER` | Nullable; updated when the project becomes active |

Rules:

- `root_path` is stored only after Rust canonicalizes it.
- deleting a project may remove Dispatch-owned metadata, but never deletes the underlying repository contents from disk
- any project-scoped foreign key may cascade from `projects`

#### `tasks`

Each task row is a persistent kanban item owned by a single project.

Required columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | Stable opaque task id |
| `project_id` | `TEXT NOT NULL` | Foreign key to `projects(id)` |
| `title` | `TEXT NOT NULL` | Card title |
| `description_markdown` | `TEXT NOT NULL DEFAULT ''` | Markdown body |
| `workflow_state` | `TEXT NOT NULL` | Lowercase storage value for the workflow enum |
| `last_run_state` | `TEXT NOT NULL DEFAULT 'idle'` | Lowercase storage value for the latest execution result |
| `last_session_id` | `TEXT` | Nullable foreign key to `agent_sessions(id)` |
| `assigned_agent_mode` | `TEXT` | Nullable; `auto` or `profile:{agent_profile_id}` |
| `markdown_export_path` | `TEXT` | Nullable path under the project root |
| `blocked_reason` | `TEXT` | Nullable explanation for `Blocked` state |
| `created_at` | `INTEGER NOT NULL` | UTC Unix epoch seconds |
| `updated_at` | `INTEGER NOT NULL` | UTC Unix epoch seconds |
| `completed_at` | `INTEGER` | Nullable completion timestamp |

Foreign-key behavior:

- `project_id -> projects(id)` uses `ON DELETE CASCADE`
- `last_session_id -> agent_sessions(id)` uses `ON DELETE SET NULL`

#### `agent_sessions`

Each session row tracks a raw terminal session, a direct CLI agent run, or an orchestrated agent session that Dispatch renders and supervises.

Required columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | Stable opaque session id |
| `project_id` | `TEXT NOT NULL` | Foreign key to `projects(id)` |
| `task_id` | `TEXT` | Nullable foreign key to `tasks(id)` |
| `source` | `TEXT NOT NULL` | Origin of the session request; see locked source values below |
| `session_kind` | `TEXT NOT NULL` | `shell`, `direct_agent`, or `orchestrated_agent` |
| `status` | `TEXT NOT NULL` | Locked runtime lifecycle state |
| `program` | `TEXT NOT NULL` | Executable path or command name |
| `args_json` | `TEXT NOT NULL DEFAULT '[]'` | JSON array of argv segments |
| `env_keys_json` | `TEXT NOT NULL DEFAULT '[]'` | JSON array of env-var names only; never secret values |
| `cwd` | `TEXT NOT NULL` | Canonical working directory |
| `transport` | `TEXT NOT NULL` | `pty` or `openclaw` |
| `exit_code` | `INTEGER` | Nullable until the session finishes |
| `started_at` | `INTEGER` | Nullable until the child actually starts |
| `ended_at` | `INTEGER` | Nullable until the session finishes |
| `created_at` | `INTEGER NOT NULL` | UTC Unix epoch seconds |
| `updated_at` | `INTEGER NOT NULL` | UTC Unix epoch seconds |

Foreign-key behavior:

- `project_id -> projects(id)` uses `ON DELETE CASCADE`
- `task_id -> tasks(id)` uses `ON DELETE SET NULL`

#### `chat_messages`

`chat_messages` is part of the initial schema even though the Chat tab lands later. This table freezes the cache shape early so Phase 8 does not invent a second persistence model.

Required columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `TEXT PRIMARY KEY` | Stable opaque message id |
| `conversation_id` | `TEXT NOT NULL DEFAULT 'main'` | v1 uses a single default conversation key |
| `project_id` | `TEXT` | Nullable project context when a message is scoped to the active project |
| `agent_session_id` | `TEXT` | Nullable related session id |
| `role` | `TEXT NOT NULL` | `system`, `user`, `assistant`, or `tool` |
| `author_kind` | `TEXT NOT NULL` | `user`, `dispatch`, `openclaw`, or `agent` |
| `body_markdown` | `TEXT NOT NULL` | Renderable message body |
| `metadata_json` | `TEXT NOT NULL DEFAULT '{}'` | Structured non-authoritative payload metadata |
| `created_at` | `INTEGER NOT NULL` | UTC Unix epoch seconds |

Foreign-key behavior:

- `project_id -> projects(id)` uses `ON DELETE SET NULL`
- `agent_session_id -> agent_sessions(id)` uses `ON DELETE SET NULL`

Rules:

- chat history is append-oriented; messages are not mutable user documents
- global chat remains possible because `project_id` is nullable
- streaming chunks may be buffered in memory, but the durable cache lands as normalized message rows

#### `settings`

`settings` stores non-secret application settings only.

Required columns:

| Column | Type | Notes |
| --- | --- | --- |
| `key` | `TEXT PRIMARY KEY` | Stable application setting key |
| `value_json` | `TEXT NOT NULL` | Canonical JSON payload for the setting |
| `updated_at` | `INTEGER NOT NULL` | UTC Unix epoch seconds |

Rules:

- secrets never land in `value_json`
- values are written only by Rust-owned commands
- frontend local state may cache settings for UX but cannot become authoritative

### Required Constraint and Index Contract

`001_init.sql` must treat the initial schema as an implementation-complete contract, not as a loose first draft.

#### Required DB-Enforced Constraints

The initial migration must enforce these constraints in SQLite:

- `tasks.workflow_state` is limited to `draft | planning | in_progress | review | done | blocked`
- `tasks.last_run_state` is limited to `idle | running | succeeded | failed | canceled | abandoned`
- `tasks.assigned_agent_mode` is limited to `NULL`, `auto`, or `profile:{agent_profile_id}`
- `agent_sessions.source` is limited to `terminal | direct_dispatch | openclaw`
- `agent_sessions.session_kind` is limited to `shell | direct_agent | orchestrated_agent`
- `agent_sessions.status` is limited to `pending | running | succeeded | failed | canceled | abandoned`
- `agent_sessions.transport` is limited to `pty | openclaw`
- `chat_messages.role` is limited to `system | user | assistant | tool`
- `chat_messages.author_kind` is limited to `user | dispatch | openclaw | agent`
- `agent_sessions.args_json` and `agent_sessions.env_keys_json` must be valid JSON arrays
- `chat_messages.metadata_json` must be a valid JSON object
- `settings.value_json` must be valid JSON

Application code still owns richer behavior and cross-field invariants, but the database is authoritative for closed value membership and JSON syntax.

#### Required Secondary Indexes

The initial migration must include these secondary indexes:

- `idx_tasks_project_workflow_updated_at` on `tasks(project_id, workflow_state, updated_at DESC, id)` for project-scoped kanban queries
- `idx_tasks_project_last_run_updated_at` on `tasks(project_id, last_run_state, updated_at DESC, id)` for project-scoped task filters and resume views
- `idx_agent_sessions_project_status_created_at` on `agent_sessions(project_id, status, created_at DESC, id)` for project session queries
- `idx_agent_sessions_task_created_at` on `agent_sessions(task_id, created_at DESC, id)` for task-linked session history
- `idx_chat_messages_project_conversation_created_at` on `chat_messages(project_id, conversation_id, created_at ASC, id)` for stable chat ordering

`projects.root_path` remains protected by its `UNIQUE` constraint, and `settings.key` remains protected by its primary key; no extra secondary indexes are required for those tables in `001_init.sql`.

### Task Workflow and Run-State Enums

Dispatch freezes two separate task state families: workflow progression and most-recent run outcome.

#### Workflow state enum

Canonical application enum:

- `Draft`
- `Planning`
- `InProgress`
- `Review`
- `Done`
- `Blocked`

Canonical SQLite storage values:

- `draft`
- `planning`
- `in_progress`
- `review`
- `done`
- `blocked`

Semantics:

- `Draft`: captured but not yet prepared for execution
- `Planning`: actively being refined into an implementation-ready task
- `InProgress`: execution is underway or expected to resume without re-triage
- `Review`: implementation is complete enough to review or verify
- `Done`: accepted and no longer active
- `Blocked`: intentionally paused on an external dependency, missing input, or unresolved constraint

`Blocked` is distinct from execution failure. A failed or canceled session updates `last_run_state`; it does not automatically force `workflow_state = blocked`.

#### Last run state enum

Canonical SQLite storage values:

- `idle`
- `running`
- `succeeded`
- `failed`
- `canceled`
- `abandoned`

Semantics:

- `idle`: no active or recent run to report
- `running`: an attached session is currently active
- `succeeded`: the latest completed run exited successfully
- `failed`: the latest completed run exited unsuccessfully
- `canceled`: the latest run was intentionally terminated
- `abandoned`: the backend lost ownership of what had previously been considered a live session

### Agent Session Source, Status, and Assignment Domains

#### `agent_sessions.source`

Canonical SQLite storage values:

- `terminal`
- `direct_dispatch`
- `openclaw`

Semantics:

- `terminal`: created from the raw "New Terminal" flow
- `direct_dispatch`: created by Dispatch launching a local CLI agent profile
- `openclaw`: mirrored from an OpenClaw-orchestrated session

#### `agent_sessions.status`

Canonical SQLite storage values:

- `pending`
- `running`
- `succeeded`
- `failed`
- `canceled`
- `abandoned`

Semantics:

- `pending`: the row exists, but the child process or external session is not yet confirmed live
- `running`: the session is live and attachable
- `succeeded`: the session ended with a successful outcome
- `failed`: the session ended unsuccessfully
- `canceled`: the session was intentionally terminated before normal completion
- `abandoned`: the backend lost ownership of a session that had previously been considered running

#### `tasks.assigned_agent_mode`

Canonical SQLite storage values:

- `NULL`
- `auto`
- `profile:{agent_profile_id}`

Semantics:

- `NULL`: no agent preference has been stored for the task yet
- `auto`: Dispatch should auto-pick the agent for this task
- `profile:{agent_profile_id}`: Dispatch should use the named agent profile id exactly

### Secret Resolution Rules

Secret lookup and persistence follow this fixed precedence:

1. `keyring 3.6.3`
2. inherited environment variable fallback
3. missing

Additional rules:

- if a keychain value exists and is non-blank, it wins
- if the keychain value is absent, Dispatch may read an inherited env var of the mapped name
- env fallback is read-only from Dispatch's perspective; Dispatch does not persist env secrets into keychain or SQLite unless the user explicitly sets them later
- secret status surfaced to the frontend is `keychain`, `env`, or `missing`
- once a secret is set, the raw value is not returned back to the frontend in later reads

## Rationale

- A five-table foundation closes the earlier gap between "chat cache is Rust-owned" and the narrower four-table placeholder lists.
- Freezing `Blocked` now avoids misusing `failed` or `review` for dependency stalls later.
- Lowercase storage strings keep SQL constraints simple while still allowing typed enums in Rust and TypeScript.
- UTC epoch seconds avoid timezone drift and keep Rust, SQL, and frontend serialization aligned.
- WAL mode plus bounded journal size is a good fit for a desktop app that mixes many small writes with long-lived reads.
- Explicit foreign keys keep the project-scoped data model coherent and reduce orphaned session or task records.
- Secret precedence belongs beside the settings contract because otherwise later implementation will be tempted to treat env vars, keychain values, and JSON settings as interchangeable sources.

## Consequences

### Positive

- Phase 1 can implement `001_init.sql` without inventing schema shape mid-flight.
- Task cards, terminal sessions, chat cache, and settings all start from one coherent Rust-owned persistence model.
- Future features can rely on fixed timestamp and enum conventions.
- The frontend has a clear contract for what is durable versus what is disposable UI cache.

### Costs

- The initial migration is broader than the bare minimum because it includes `chat_messages` before the Chat tab ships.
- Adding `Blocked` means later task-transition logic must treat workflow state and run state as separate concerns.
- JSON payload columns such as `args_json`, `env_keys_json`, `metadata_json`, and `value_json` require disciplined serialization rather than ad hoc strings.

### Follow-on Constraints

- A later history migration may add `save_points`, but history metadata is not part of this initial schema.
- Any future schema that wants per-project settings, agent registry tables, or richer chat threading must extend these tables through migrations rather than bypassing them with a plugin store or ad hoc files.
